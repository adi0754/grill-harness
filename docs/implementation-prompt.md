# Grill Harness 实现提示词

你现在要创建并实现一个新的、可长期维护的软件工程工作流 Skill：`grill-harness`。

这不是设计讨论。产品设计已经完成并由用户批准。不要重新发起无边界需求访谈，也不要只输出方案、伪代码或脚手架。请创建 GitHub 仓库，完成实现、测试、独立审查和推送。

## 权威输入

开始前完整阅读以下两个文件：

1. 设计规格：
   `<SOURCE_DESIGN>`
2. 实施计划：
   `<SOURCE_PLAN>`

设计规格是产品和行为合同，实施计划是执行顺序。若两者冲突，以设计规格为准，并在报告中记录冲突。若文件不存在或无法完整读取，停止并要求用户重新提供，不要根据本提示词猜测缺失设计。

仓库规则：

- 仓库名：`grill-harness`；

## 必须遵守的实现方式

1. 先调查当前 Agent Skills 规范、skills.sh CLI、Codex 和 Claude Code 的安装结构，以及已安装 Skill 示例。
2. 阅读 `mattpocock/skills` 当前最新实现，记录实际 commit、许可证、相关 Skill 文件和行为差异。
3. 按设计规格只发布一个公开 Skill：`grill-harness`。
4. 所有运行时状态和文档只能写入 `~/.grill-harness/`；不得向用户项目仓库写入 Grill Harness 指针、状态或文档。
5. 第一版对 Codex 和 Claude Code进行一级验证；其他 skills.sh Agent 只能标记为未完成同等级验证。
6. 必需上游能力是 `grilling`、`domain-modeling`、`codebase-design`。
7. `grill-with-docs` 只作为兼容性参考，不作为内部可调用依赖。
8. 不复制上游 Skill 文本，不静默替代缺失的必需能力，不未经用户同意安装或更新第三方 Skill。
9. Grill Harness 不自动选择或启动正式实施、审查、修复、集成和验收 Agent。它生成本地任务包和短启动提示词，由用户选择本地模型和 Agent。
10. 不实现网页模型便携提示词，不实现跨设备同步，不实现完整外部 Loop。
11. 人读运行时目录、文件名和内容以简洁中文为主；稳定机器协议文件使用设计规定的英文名称。
12. 使用确定性脚本处理状态、原子写入、迁移、DAG、Frontier、冲突、证据、依赖检测和上游比较；不要把可机械验证的规则只写成提示词。
13. 优先使用 Python 标准库和 JSON-compatible YAML，避免为基础功能引入新的第三方运行时依赖。若确实需要安装依赖，先请求用户授权。
14. 不修改用户全局 Agent 配置，不删除或覆盖已有 Skill，不执行破坏性 Git 操作。

## Skill 开发质量门禁

把 Skill 开发视为针对 Agent 行为的 TDD：

1. 在写正式 `SKILL.md` 前，先为以下失败模式创建压力场景并运行无 Skill 基线：
   - 未确认需求就编码；
   - 跳过路线选择；
   - 审查者只相信实施总结；
   - 没有真实证据就宣称完成。
2. 保存基线 Agent 的真实行为和合理化借口。
3. 编写最小 Skill 和阶段协议以修复已经观察到的失败。
4. 在新上下文中重复相同场景，验证行为改善。
5. 发现新漏洞时只增加针对性规则，并重新测试。

不得以“提示词看起来清楚”代替前向测试。

## 工作顺序

严格执行实施计划中的任务顺序：

1. 安全创建私有 GitHub 仓库并保存设计基线；
2. 完成无 Skill RED 基线；
3. 创建 skills.sh 可发现的单 Skill 结构；
4. 实现用户目录、项目识别、状态和迁移；
5. 实现 Ledger、产物校验和变更传播；
6. 实现任务 DAG、Frontier、并行冲突和证据协议；
7. 实现依赖预检和上游适配层；
8. 实现中文模板、角色任务包和本地启动提示词；
9. 编写精简的主 `SKILL.md` 路由器；
10. 在 Codex 和 Claude Code 中运行场景验证；
11. 在隔离环境中验证 skills.sh 安装和更新；
12. 进行独立 Standards 与 Spec 审查；
13. 修复问题、重新验证、提交并推送。

按实施计划频繁提交。不要把所有工作堆在一个最终提交中。

## 用户控制与暂停条件

实现过程中自主推进，不要等待用户逐步确认。但是出现以下情况必须暂停：

- GitHub 未登录或权限不足；
- `grill-harness` 远程仓库或本地目标目录已经存在；
- 需要将仓库设为 public；
- 需要安装新的第三方依赖或第三方 Skill；
- 需要修改全局 Agent 配置；
- 需要删除、覆盖已有 Skill 或用户文件；
- 需要执行破坏性 Git 操作；
- 设计规格存在无法合理解决的内部矛盾；
- Codex 或 Claude Code 验证需要当前环境没有的授权。

普通实现选择、文件结构细节和测试修复不需要询问，只要不违背设计规格。

## 完成标准

只有在以下条件全部满足后才能宣称完成：

- 私有 GitHub 仓库已经创建；
- 实现已提交并推送；
- skills.sh 能发现 `grill-harness`；
- README 中的安装命令经过实际验证；
- 运行时数据只写入 `~/.grill-harness/` 或测试使用的隔离替代目录；
- 三个用户硬检查点和轻量模式行为有场景证据；
- 本地任务包可由新 Agent 会话读取和执行；
- 缺失上游能力时正确阻塞并给出安装指导；
- 证据不足时不会无条件验收通过；
- 上游检查不会自动覆盖本地行为；
- Codex 和 Claude Code 的验证状态被准确报告；
- 独立审查中的阻塞问题已经修复或明确保留为限制。
