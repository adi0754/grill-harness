# Grill Harness 设计规格

日期：2026-07-11
状态：已确认并实现，待合并
目标：设计一个单入口、可恢复、可升级、跨本地 Agent 的软件工程工作流 Skill。

## 1. 产品定位

Grill Harness 是一个以文件契约和持久化状态驱动的个人软件工程工作流控制台。

它解决以下问题：

- Agent 在需求和路线稳定前过早编码；
- 用户只看到一条方案，失去路线选择权；
- 同时生成多份完整方案导致 Token 浪费；
- 精确约束在需求、规格、任务和实现之间丢失；
- 设计没有经过真实仓库挑战；
- 大任务没有按可验证的垂直切片拆分；
- 不安全的并行任务修改共享文件或契约；
- 实施、审查和验收共享同一长上下文；
- 不同模型只能依赖上一模型的总结；
- 没有真实测试证据就宣称完成；
- 工作流中断后无法恢复；
- 项目结束后没有保留决策和可复用知识；
- 上游 Skills 更新后难以判断是否兼容。

Grill Harness 按角色组织工作，不绑定某个模型或厂商。它只生成本地任务包和启动提示词，由用户决定使用哪个模型和 Agent。

## 2. 已确认的产品决策

### 2.1 单一公开入口

只发布一个 Skill：

```text
grill-harness
```

开始、继续、状态、恢复、上游检查、拆分、审查和验收均作为内部模式，由主 Skill 根据用户意图和持久化状态路由。

用户不需要记忆 `grh-start`、`grh-spec`、`grh-review` 等阶段名称。

### 2.2 存储位置

所有运行时状态、过程文档和最终文档只写入：

```text
~/.grill-harness/
```

项目仓库内不写入 Grill Harness 指针、状态或文档。

Skill 安装文件仍位于各 Agent 的标准 Skill 目录；`~/.grill-harness/` 仅保存运行时数据。

### 2.3 运行时兼容

第一版一级验证目标：

- Codex；
- Claude Code。

其他 skills.sh 支持的 Agent 可以安装公共 Skill，但必须标记为未完成同等级验证。

### 2.4 安装方式

Grill Harness 作为独立 Git 仓库发布，并支持通过 skills.sh 全局安装。

安装命令必须在实现阶段使用当前 CLI 实际验证。README 只展示已验证命令，不把设计期示例当作最终命令。

### 2.5 上游依赖

必需运行时能力：

- `grilling`；
- `domain-modeling`；
- `codebase-design`。

`grill-with-docs` 继续作为重要上游兼容性参考，但不作为内部运行时调用入口。原因是当前上游将其定义为仅用户调用的顶层 Skill，内容只是组合 `grilling` 和 `domain-modeling`。

可选增强能力：

- `research`；
- `prototype`；
- `tdd`；
- `code-review`。

设计参考但不作为运行时依赖：

- `wayfinder`；
- `to-spec`；
- `to-tickets`；
- `implement`；
- `handoff`。

Grill Harness 不复制上游 Skill，不静默模拟缺失的必需能力，不未经用户许可安装或更新第三方依赖。

### 2.6 多模型协作

Grill Harness 不自动选择或启动正式实施、审查和验收模型。

它为每个角色生成：

- 本地任务包；
- 本地启动提示词；
- 输出契约。

用户在选定的本地 Codex、Claude Code 或其他能访问本机文件的 Agent 新会话中发送启动提示词。

第一版不生成网页模型便携提示词，不实现跨设备同步，也不实现完整的第三方 Loop 调度器。

## 3. 上游工作流基线

调查基准：`mattpocock/skills` 默认分支 `main`，2026-07-10 的 commit `391a2701dd948f94f56a39f7533f8eea9a859c87`。

当前上游不是固定单线工作流，而是主链、情境入口和底层能力组合：

```text
普通需求
  → grill-with-docs
  → [需要实际验证时：handoff → prototype → handoff 返回]
  → [多会话：to-spec → to-tickets]
  → implement
  → 内部使用 tdd
  → 内部运行 code-review
  → commit

巨大且模糊的工作
  → wayfinder
  → research / prototype / grilling + domain-modeling / task
  → 路线清楚后进入 to-spec 或 implement
```

关键事实：

- `wayfinder` 是情境入口，不是所有工作的默认起点；
- `research`、`prototype` 和 `grilling` 是按问题选择的探索方式，不是固定顺序；
- `domain-modeling` 和 `codebase-design` 是底层纪律，不是线性阶段；
- `tdd` 当前采用 red → green，重构位于 review 阶段；
- `to-tickets` 生成垂直切片、阻塞边和 Frontier，但不负责外部调度；
- `handoff` 是跨会话桥梁，不是固定收尾步骤。

Grill Harness 在此基础上增加持久化状态、路线控制、稳定 ID、仓库挑战、角色任务包、集成验收和上游兼容机制。

## 6. 项目识别与多工作流

项目身份由以下信息组合：

- 规范化绝对路径；
- 规范化 Git remote；
- Git 根提交或其他稳定仓库特征；
- 项目简称；
- Grill Harness 项目 ID。

仓库移动后优先通过 remote 和仓库特征重新关联。出现多个候选时必须让用户选择。无法可靠识别时创建新项目记录，不根据名称自动合并。

非 Git 项目以路径为主。路径变化后需要用户确认重新关联。

同一项目可以有多个活动工作流。只有一个活动工作流时自动继续；存在多个时展示列表；没有时询问是否创建。

## 7. 事实权威模型

不同产物承担不同权威职责：

1. 真实仓库：当前代码、配置和文件事实；
2. 决策账本：用户确认的需求、约束和决策；
3. 当前规格：已批准的实施合同；
4. 任务图和任务包：当前授权范围；
5. 测试与验收证据：是否真正完成的依据；
6. `state.yaml`：产物索引和流程位置，不覆盖真实产物。

发生冲突时暂停受影响阶段，列出冲突来源，以真实仓库确认现状，并通过变更控制更新决策、规格和任务。

## 8. 工作流状态机

### 8.1 阶段

```text
开始或继续
  → 预检
  → 对齐
  → 用户确认需求基线
  → 设计
  → 用户选择路线
  → 仓库挑战与规格深化
  → 用户批准最终规格
  → 拆分与调度
  → 实施循环
  → 独立保障
  → 知识沉淀和归档
```

### 8.2 三个用户硬检查点

1. 确认需求基线；
2. 选择路线；
3. 批准最终规格。

轻量模式可以把三个检查点合并为一次确认，但不得零确认编码。

### 8.3 状态集合

```text
pending
in_progress
needs_user
blocked
completed
stale
superseded
skipped
failed
cancelled
```

`completed` 必须存在产物和证据。上游决策变化后相关产物变为 `stale`。可选阶段才能 `skipped`，并记录原因。

### 8.4 运行模式

轻量模式：局部 Bug、配置或小范围修改，保留仓库检查、一次授权、真实验证、diff 检查和最小验收。

标准模式：默认模式，使用需求基线、路线选择、仓库挑战、规格、任务包、审查和验收。

Wayfinding 模式：用于巨大且一个会话无法看清的工作，每个新会话只解决一个调查或决策问题，路线清楚后进入标准流程的对应阶段。

Harness 推荐模式并说明理由，用户可以覆盖。

## 9. 变更控制

最终规格批准后发现新事实，创建 `CHG-xxx` 记录，不直接修改规格继续开发。

变更记录包含：

- 新事实或冲突；
- 来源证据；
- 受影响的 REQ、DEC 和 TASK；
- 是否影响路线；
- 已完成代码是否需要重验；
- 建议退回阶段。

退回规则：

- 目标或范围变化 → 对齐；
- 路线失效 → 路线选择；
- 仓库事实或技术设计变化 → 规格；
- 任务边界错误 → 拆分与调度。

用户批准变更后才更新 Ledger、规格和任务图。

## 10. 文档与追踪协议

### 10.1 稳定 ID

```text
REQ-001   需求
DEC-001   决策
CON-001   约束
RISK-001  风险
CHG-001   变更
TASK-001  任务
ISSUE-001 审查问题
EVD-001   证据
```

所有记录使用少量公共字段，仅在相关类型中增加专用字段，避免为每条记录填充大量无关内容。

关键要求：

- 不静默覆盖；
- 修改时增加版本；
- 保留否定要求、数值、默认值、边界和例外；
- 标记替代关系；
- 标记受影响的规格、任务和证据。

### 10.2 路线卡

路线明确且低风险时只生成一张推荐路线卡；存在实质分歧时生成两至三张。

每张卡只包含核心思路、主要改动边界、关键决策组合、优势、代价、最大风险、适用条件、速度、长期影响和推荐程度。

用户选择前不生成多份完整规格。未选路线只保存在过程产物中，不进入任务上下文。

### 10.3 仓库挑战

仓库挑战角色必须检查真实代码、调用方、同步和异步路径、复用能力、测试体系、迁移、安全、并发、事务和配置，并引用真实路径和符号。

结论分类：成立、部分成立、不成立、遗漏、需要用户决策、无法验证。

### 10.4 任务图

任务使用垂直切片，宽范围机械改造使用 expand–migrate–contract。

`tasks.yaml` 至少记录：

- 任务 ID；
- 状态；
- 依赖；
- 并行组；
- REQ/DEC/CON；
- 允许和禁止修改范围；
- 共享契约；
- Git 基线；
- 任务包和输出路径。

Frontier 由依赖满足情况计算。写入范围只是调度预测，任务开始前和完成后都要检查真实 diff。

## 11. 本地角色任务包

每个角色生成一份任务包和一份短启动提示词：

```text
TASK-001-实施任务.md
TASK-001-启动提示词.md
TASK-001-审查任务.md
TASK-001-审查启动提示词.md
TASK-001-修复任务.md
最终集成任务.md
最终验收任务.md
```

任务包包含：

- 当前角色和目标；
- 项目工作目录；
- Git 基线；
- 必须读取的文件；
- 相关需求和决策；
- 已确认仓库事实；
- 允许和禁止修改范围；
- 工作步骤；
- 测试和证据要求；
- 验收标准；
- 停止条件；
- 输出路径和格式。

启动提示词只要求新 Agent 读取任务包、检查真实仓库、执行任务并写回报告。

任务包不依赖原始聊天历史。第一版不生成网页模型便携提示词。

## 12. 本地多模型协作

用户决定将哪个任务包交给哪个模型或 Agent。

Grill Harness 可以描述任务所需能力和独立性要求，但不推荐或强制某个品牌模型。

正式角色不能直接把系统状态改为最终通过。它们只写自己的报告，主 Grill Harness 会话负责检查真实仓库、diff 和证据后更新状态。

独立性必须明确区分：

- 新会话；
- 不同 Agent；
- 不同模型；
- 不同模型厂商。

启动了子 Agent 不等于完成跨模型验证。

## 13. Git 基线与并行隔离

每个任务记录基线 Commit、目标工作区、允许范围、预计写入范围、实际 diff、合并目标和顺序。

并行任务只有在使用不同 worktree 或独立工作区、独立分支且不存在共享契约冲突时才可执行。

未建立独立 worktree 时，可并行只是一项分析结论，不能让两个 Agent 同时修改同一工作区。

如果环境不适合 worktree，自动降级为串行任务提示词。

Grill Harness 不自动执行破坏性 Git 操作。创建分支或 worktree 的方式进入调度说明，由用户确认或交给已授权的本地 Agent。

## 14. 实施、审查和验收

### 14.1 实施

实施者读取规格、任务包和真实仓库，只完成当前任务，按任务性质决定是否使用上游 `tdd`，实际运行测试、类型检查和构建，检查 diff，并输出需求到证据映射。

发现规格冲突、重大依赖、越权修改或新高风险产品决策时必须停止。

实施完成后状态只能进入等待审查。

### 14.2 独立审查

审查默认只读产品代码，使用两条独立轴：

- Standards：正确性、安全性、可维护性、性能、错误处理、测试质量和代码库规范；
- Spec：需求覆盖、决策覆盖、非目标、范围、兼容性和未验证假设。

审查者必须检查真实完整文件和 diff，不能只读实施报告。

### 14.3 修复和复审

修复角色逐条验证问题，对成立问题做最小修复，对不成立问题提供证据。阻塞和高风险问题修复后必须复审。

### 14.4 集成检查

并行任务分别通过审查后，检查合并冲突、公共契约、数据和配置、重复实现、跨任务矛盾、生成文件、迁移顺序、跨模块流程、集成测试和全局需求覆盖。

### 14.5 最终验收

最终验收使用新的本地 Agent 会话，不能使用实施会话。验收者从需求基线和最终规格重新理解目标，检查最终仓库、完整 diff、未提交文件，并实际运行可执行命令。

结论只能是：验收通过、有条件通过、验收不通过、无法验证。

没有有效证据时不得无条件通过。

## 15. 证据协议

`evidence.yaml` 记录：

- 证据 ID；
- 对应需求、决策、任务和问题；
- 实际命令；
- 工作目录；
- 执行时间；
- 退出码；
- 输出位置；
- Git 基线；
- 产生者；
- 可复现性；
- 当前有效性。

长期 Markdown 只保存摘要和原始证据路径，不复制全部日志。

没有有效证据的 REQ 或 DEC 不得无条件验收通过。

## 16. 知识沉淀

最终生成 `最终产物/项目经验.md`，只保存：

- 最终目标和结果；
- 选中和否决路线及原因；
- 关键决策；
- 重要仓库事实；
- 实施偏差；
- 重要审查发现；
- 验收结果；
- 未解决风险；
- 可复用经验；
- 下次应避免的问题。

不保存完整对话、临时推理或大段测试日志。

## 17. 依赖预检

预检结合：

- `npx skills list --json`；
- Agent 标准 Skill 目录；
- Skill 元数据；
- 符号链接目标；
- 当前 Codex 或 Claude Code 运行时能力。

必需能力缺失时：

- 列出缺失项和用途；
- 阻止进入依赖该能力的阶段；
- 展示经现场验证的安装命令；
- 不自动安装；
- 安装完成后重新检查。

可选能力缺失时记录未使用的增强能力，并使用 Grill Harness 自己的角色协议执行相应可选阶段。

## 18. 上游持续升级

`上游清单.yaml` 记录：

- 仓库和跟踪 ref；
- 最近检查 Commit 和时间；
- 许可证；
- 能力名称；
- 上游文件；
- 内容哈希；
- 使用的行为契约；
- 本地差异；
- 兼容风险；
- 最近兼容测试结果。

上游检查流程：

1. 检查本地依赖和可加载状态；
2. 比较上次 Commit 和内容哈希；
3. 网络可用时检查远程更新；
4. 只读取相关 Skill、文档、Changelog 和 Release；
5. 分类新增、修复、行为变化、元数据变化、重命名删除和设计冲突；
6. 判断是否触碰能力契约；
7. 运行固定兼容场景；
8. 生成中文兼容报告；
9. 建议无需处理、更新依赖、修改适配器、暂缓更新或人工决策；
10. 用户批准后才更新；
11. 更新后重新检查和测试。

不自动覆盖本地实现，不自动接受上游所有变化。

## 20. 状态版本和迁移

机器文件包含 `schema_version` 和 `workflow_version`。

迁移规则：

1. 检查迁移器；
2. 创建备份；
3. 在临时目录迁移；
4. 验证引用和 ID；
5. 原子替换；
6. 写入迁移报告；
7. 失败时保留原状态并停止。

不静默丢弃未知字段，不无备份修改状态，不因上游更新重写用户决策。

## 21. 测试策略

- 任务包完整性；
- 证据有效性；
- 状态和产物对账；
- 敏感信息扫描；
- 上游变化分类；
- 状态迁移和回滚。

### 21.2 场景前向测试

在 Codex 和 Claude Code 的新会话中分别验证：

- 局部 Bug 的轻量模式；

- Wayfinding 模式；

- README 命令与实际行为一致。

## 22. 错误处理和安全

覆盖以下错误：

- 必需依赖缺失；
- skills CLI 不存在；
- 网络或上游不可用；
- 状态文件损坏；
- 项目路径变化；
- 多个项目候选；
- Git 基线漂移；
- 工作区存在用户未提交改动；
- 任务报告缺失；
- 验证命令无法运行；
- 证据过期；
- 实际修改超出预测范围。

错误信息必须使用中文，说明发生了什么、影响、已完成内容、不会自动执行什么以及恢复方式。

禁止：

- 自动执行 `git reset --hard`；
- 自动强制推送；
- 自动删除未跟踪文件；
- 覆盖用户未提交工作；
- 未授权安装或更新第三方 Skill；
- 保存凭据；
- 把推测写成事实；
- 绕过用户检查点；
- 在证据不足时宣称完成。

## 23. 第一版明确不做

- 不自动选择模型；
- 不自动启动正式实施、审查或验收 Agent；
- 不支持网页模型提示词导出；
- 不实现跨设备同步；
- 不向项目仓库写 Grill Harness 文件；
- 不实现完整外部 Loop；
- 不复制上游 Skill；
- 不自动安装或更新第三方依赖；
- 不保证所有 skills.sh Agent 完成同等级测试；
- 不为小任务生成完整重型文档；
- 不自动执行破坏性 Git 操作。

第一版保留稳定 `tasks.yaml` 协议，为未来外部 Loop 和更多运行时适配预留扩展点。

## 24. 验收结果

实现完成后至少证明：

- 用户只需调用一个 Skill；
- Codex 和 Claude Code 能通过 skills.sh 安装；
- 运行时文件只进入 `~/.grill-harness/`；
- 缺少必需上游能力时正确阻塞和引导；
- 标准流程在三个用户检查点暂停；
- 轻量模式不会生成重型产物；
- 路线选择前不深化多份方案；
- 决策、规格、任务和证据可追踪；
- 仓库挑战使用真实文件；
- 垂直任务图和 Frontier 可恢复；
- 并行任务在独立工作区执行；
- 本地启动提示词可在新 Agent 会话使用；
- 审查和最终验收检查真实 diff 和命令；
- 证据不足时不判定通过；
- 上游变化生成兼容报告且不自动合并；
- 状态迁移可备份、验证和回滚。
