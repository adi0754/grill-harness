# 修复任务

## 给用户的话

### 30 秒结论：只修复已经证实的问题

修复者会先验证审查发现是否成立，再对成立问题做最小改动。不会借修复之名改变已批准规格或扩大产品范围。每次修复都要留下可复现证据并等待独立复审。同一实现问题连续第三次失败时停止普通修复并交回恢复流程。

## 修复内容

- 角色与目标：逐条验证问题并作最小修复
- 修复历史：逐轮动作、结果和证据
- 分支 / worktree / 并行组：
- 共享契约、迁移和生成物：
- 允许修改范围：
- 禁止修改范围：
- 预计写入路径：
- 停止条件：问题要求越权、改变已批准规格、引入新风险，或同一实现问题第三次仍失败。第三次后禁止普通修复任务，停止并建议用户进入恢复流程；不得自动换路、补证据或修改轮次阈值。

## 证据与标识

- 任务与问题：`TASK-001` / `ISSUE-001`
- 项目工作目录（绝对路径）：
- 当前工作流目录（绝对路径，仅限 `~/.grill-harness/`）：
- Git 基线：
- 失败分类：`implementation_failure | route_failure | evidence_failure | workflow_integrity_failure`
- 稳定指纹：`FAIL-*`（只由 ISSUE ID、失败验收/命令和 Git 基线生成）
- 当前修复轮次：`1 | 2 | 3+`
- 第三次同一实现失败后的建议入口：`grh-recover`
- 失败证据：失败验收项、实际命令/退出码、原始输出路径和当前性
- Review 契约：`review_required: true`；完成前必须通过 `task-review` 持久化审查事实与逐项证据
- 输入产物（逐项绝对路径与版本/基线）：规格、审查报告、完整相关文件
- 输出路径（绝对路径）：`<当前工作流>/过程产物/审查修复/TASK-001-修复报告.md`
- 输出格式：成立/不成立证据、实际 diff、测试证据、待复审问题
