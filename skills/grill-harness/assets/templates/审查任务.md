# 审查任务

## 给用户的话

### 30 秒结论：分别检查代码质量和需求实现是否正确

审查者会保持产品代码只读，并从代码规范与需求合同两条轴独立检查。每条发现都要给出真实路径、证据和处置分类，不能只写一份笼统报告。两条轴不会互相抵消。输入不完整或基线不符时会停止并说明无法审查。

## 审查内容

- 角色与目标：独立审查者
- 被审分支 / worktree / 并行组：
- 共享契约、迁移和生成物：
- Standards 轴：正确性、安全、维护性、性能、错误处理、测试质量、仓库规范
- Spec 轴：需求与决策覆盖、非目标、范围、兼容性、未验证假设
- 结构化 comments：每条包含 `axis: standards|spec`、分类（blocking / must_fix / optional / invalid）、路径或符号证据和处置状态
- 允许修改范围：默认只读产品代码
- 禁止修改范围：未获授权的产品修改
- 预计写入路径：仅当前工作流中的审查报告
- 停止条件：基线不符、输入缺失、无法读取完整 diff

- 项目工作目录（绝对路径）：
- 当前工作流目录（绝对路径，仅限 `~/.grill-harness/`）：
- Git 基线：
- 输入产物（逐项绝对路径与版本/基线）：任务包、规格、实施报告、完整文件、真实 diff
- comments 关联：稳定 `ISSUE-*`、当前 `EVD-*` 和处置状态
- 登记要求：通过 `task-review` 追加写入任务 `review_history`；不得只写散文报告或保留 `comments: []`
- 输出路径（绝对路径）：`<当前工作流>/过程产物/审查修复/TASK-001-审查报告.md`
- 输出格式：逐轴结论、结构化 comments 与稳定 `ISSUE-001`，附路径、符号和证据
