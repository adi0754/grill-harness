## 共同执行规则

每个阶段开始前：

1. 读取 `status`/`reconcile` 的真实结果和当前阶段输入产物。
2. 检查输入版本、Git 基线、稳定 ID 和门禁；不从聊天摘要补齐缺失事实。
3. 只把产物写到当前工作流目录。除获准实施的角色外，产品仓库保持只读。
4. 将新增或变化的需求、决策、约束、风险和变更写入决策账本，并建立替代关系，不覆盖历史。
5. 发现基线漂移、规格冲突、新高风险决策或越权范围时停止，登记 `CHG-xxx`，按“用户决策协议”交回用户。
6. 每个公开入口只执行它声明的范围；完成后只建议下一入口，不得自动串联或偷偷改道。用户指定更窄范围时按更窄范围停止。

系统记录的唯一写入口：

- `record --kind ledger|artifact|task|evidence --record <记录文件> [--project <产品仓库>]`：校验后同步 state 与独立清单；不使用一步式批准时，门禁批准前先登记真实 `DEC/CHG` Ledger 记录，登记 evidence 时必须传产品仓库以核对当前 Git 基线；
- `approve --gate <门禁> --approval-id <DEC/CHG> --artifact-version <ID=版本> [--decision-record <DEC/CHG记录文件>]`：绑定真实当前产物；携带 `--decision-record` 时在一次事务中完成 Ledger 记账与门禁绑定，不携带时保持先 `record` 后 `approve` 的兼容流程；
- `transition --phase <阶段> --to <状态> [--artifact <ID>] [--evidence <ID>]`：执行合法迁移；v1 的十二个主阶段都是必需阶段，不能在运行时改成 optional/skipped，`cancelled/superseded` 也不算完成前序。研究或原型是阶段内部的可选产物，不是可跳过的主阶段；
- `task-transition --task <TASK-ID> --to <状态> --project <产品仓库> [--evidence <EVD-ID>]`：按合法生命周期推进任务；每次都重新核对所属项目和当前含脏状态的 Git 基线，新任务只能以 `pending/current` 登记，完成时必须绑定当前基线证据和已存在的实施报告；
- `invalidate-chain --workflow <工作流> --change-id <CHG-ID>`：仅在 `grh-recover` 中使用；读取用户已批准 CHG 的连续 `affected_phases`，原子将最早阶段置为 `stale`、其后阶段重置为 `pending`、撤销受影响门禁并同步失效相关任务、产物和非共享证据；保留失效前快照，禁止用它绕过新的人工门禁；
- `migrate --workflow <工作流>` 与 `rollback --report <迁移报告>`：带备份迁移或回滚。

运行 `python3 <grh> <command> --help` 查看参数。命令失败时保留 JSON 和原文件，不绕过校验手工推进。

人工门禁统一使用以下批准序列：形成待批准产物时先预分配本轮 `DEC/CHG` ID，并把该 ID 写入产物记录的 `decisions`；用户对这个真实产物版本作出明确确认后，才创建批准记录并绑定门禁。向用户提出门禁问题时使用 `assets/templates/门禁提问.md` 的格式。优先用带 `--decision-record` 的一步式 `approve` 在同一事务中完成后两步；若沿用两条命令，仍必须先 `record --kind ledger`、再 `approve`，不得倒序或手工改文件。

用于人工门禁的 Ledger 记录必须明确包含 `status: approved`、`approved_by: user`、对应 `gate`、与批准命令完全一致的 `artifact_versions`，以及非空的 `user_reply_verbatim`（用户答复原文）或 `no_question_reason`（无需提问理由）之一；`approve` 会拒绝缺少问答证据的门禁批准。`id` 必须与 `--approval-id` 一致，且目标产物记录的 `decisions` 已包含该 ID。记录只有在本轮用户真实确认后才能创建，不能把 Agent 建议、旧摘要或普通 `decisions` 引用伪装成批准。

## 面向用户的沟通协议

公开入口首次汇报、等待用户门禁和结束交接时，先输出“30 秒结论”，最多五句话：

1. 当前要解决的业务或工程问题；
2. 当前阶段、状态和最重要的已确认事实；
3. Harness / Agent 正在负责什么；
4. 用户现在需要决定或执行什么；
5. 推荐下一步、原因，以及当前不能宣称的结论。

机器 JSON、稳定 ID、产物版本和命令是证据，不是主要叙事。先用自然语言说明影响和决策，再给精确标识、路径或命令。不得只回复“当前阶段是 requirements_baseline”“请批准 DEC-001”或“建议继续 grh-plan”，而不解释这代表什么、为什么现在需要用户、跳过会有什么风险。

人机职责保持稳定：

- Harness 维护项目身份、状态机、门禁、版本、授权范围和入口建议；
- Agent 读取真实仓库、澄清问题、查证事实、形成产物，并只执行当前授权范围；
- 用户确认目标与非目标、作产品或风险取舍、选择路线、批准最终规格、选择各角色使用的模型或 Agent，并作最终验收决定。

介绍整个流程时优先使用 `grh-start → grh-plan → grh-run → grh-check → grh-learn` 的主线，再把 Router、恢复和上游检查作为辅助入口说明；不要先让用户记忆八个入口。所有流程连线只表示允许方向，不表示自动执行。

### 黑话对照

对用户先使用下表中的明白中文；机器词只放在随后证据中：

| 机器词 | 对用户的中文说法 |
|---|---|
| `preflight` / `alignment` / `requirements_baseline` | 环境是否可用 / 先对齐目标 / 确认我们理解的是同一个需求 |
| `design` / `route_selection` / `repository_challenge` | 验证关键未知项 / 选择实施路线 / 用真实仓库挑战路线假设 |
| `specification` / `final_spec_approval` / `tasking` | 写成唯一最终规格 / 确认这份实施合同足以授权写代码 / 拆成可验证任务 |
| `implementation` / `independent_assurance` / `knowledge_archive` | 逐任务实施 / 独立审查与验收 / 在验收后沉淀经验 |
| `requirements_baseline` 门禁 | 需求理解确认 |
| `route_selection` 门禁 | 路线选择确认 |
| `final_spec_approval` 门禁 | 实施合同授权 |
| `PASS_WITH_LIMITATION` | 有条件通过；必须同时说清限制 |
| `stale` | 这份事实已经失效，需要重新核对 |
| `owner baseline` | 工作流所属仓库的当前基线；多仓时通常是聚合根 |
| `must_fix` | 当前交付前必须修复的问题 |

## 用户决策协议

模型以为的理解不等于用户想要的。任何把问题交给用户决定的时刻都执行同一协议：三个人工门禁、对齐取舍、调查 Agent 选择、仓库挑战判出的“需用户决策”假设、实施或修复中登记的 `CHG-xxx`、恢复时的候选权威版本与新路线、验收限制的接受、知识两步晋升。

1. 事实自查：仓库、既有产物或工具能回答的不问用户；每个问题都要说得出为什么只能由人决定。
2. 实际调用 `grilling` 并遵循其契约：一次只问一个问题，等待答复后再问下一个；按依赖顺序走决策树，先问会改变后续问题的决策；不得一次抛出问题清单。
3. 每个问题附推荐答案、理由、影响和代价。向用户提出门禁或决策问题时使用 `assets/templates/门禁提问.md` 的格式。
4. 答复原文写入 `过程产物/需求审问/用户确认记录.md`，并绑定结果 `DEC/RAD/CHG` ID；不得把 Agent 转述、推荐项或“看起来同意”当作用户原话。
5. 某个阶段或门禁整段无需提问时，在同一记录写明所属阶段、理由与事实依据；无问记理由与逐问原文同等重要。

“实际调用”的运行时语义：Claude Code 中通过 Skill 工具真实调用 `grilling`；Codex 等以文件为载体的运行时读取 `preflight` 返回的已验证安装路径下的 `SKILL.md` 并遵循其契约。两者都不得凭记忆复述或假装已调用；`grilling` 缺失时按必需能力规则失败关闭。

## 预检与对齐

运行 `identify`、`preflight`、`status`。必需能力为：

- **REQUIRED SUB-SKILL:** 使用 `grilling` 进行逐问式需求审问。
- **REQUIRED SUB-SKILL:** 使用 `domain-modeling` 维护领域语言和真正值得记录的决策。
- **REQUIRED SUB-SKILL:** 使用 `codebase-design` 挑战模块深度、接口、seam、adapter、测试面和 locality。

缺少任何必需能力时停止在预检，列出用途和当前 CLI 生成的安装建议；未经用户批准不得安装。`grill-with-docs` 只作兼容参考，不作为运行时调用入口。

`research`、`prototype`、`tdd`、`code-review`、`wayfinder`、`to-spec`、`to-tickets`、`implement` 只作可选增强；缺失时按 `能力编排矩阵.md` 使用 GRH 自研产物，不影响 `ready`，也不得假装已经调用。

初步侦察只读目标仓库，至少确认：仓库规则、语言与构建系统、入口、相关模块/调用方、测试命令、Git 状态和可疑的共享契约。此时不提出假装已验证的技术方案。

先与用户对齐工作流目标、成果形态、范围、风险容忍、时间约束和运行模式：

- **轻量**：低风险、局部、可在一个角色任务中闭环；精简产物，但需求基线、路线选择、最终规格三个门禁仍分别批准。
- **标准**：默认；完整使用路线、挑战、拆分、独立保障。
- **Wayfinding**：方向巨大或关键事实无法放进一个会话；先生成互相独立的调查/决策任务包，每个新会话只回答一个问题，收敛后回到标准流程。

## 自适应需求雷达

在形成需求基线之前创建并持续更新：

- `过程产物/需求审问/需求雷达.md`：覆盖需求澄清、需求遗漏、需求牵连、需求悖论和相似实现五类信号；
- `过程产物/需求审问/问题与发现.yaml`：用 `RAD-xxx`、阻塞级别、证据和处置状态建立机器可追踪记录；
- `过程产物/需求审问/相似实现对照.md`：记录搜索范围、真实路径与符号、相同/差异、公共契约、可复用测试和不可复用原因。

对用户汇报时，把五维雷达翻译成以下问题，不直接把分类名或 `RAD-xxx` 当作结论：

| 雷达维度 | 向用户解释的问题 | Agent 的查证责任 |
|---|---|---|
| 需求澄清 | 谁在什么条件下，要得到什么结果？ | 查清角色、触发、目标、非目标、边界和歧义 |
| 需求遗漏 | 正常路径之外，失败、权限、空状态、兼容和回滚怎么办？ | 补齐异常、迁移、运维和验收信号 |
| 需求牵连 | 这次改动会沿调用方、数据和系统边界影响到哪里？ | 追查接口、状态、事件、配置、部署和文档 |
| 需求悖论 | 目标、约束、成本、期限和验收是否能同时成立？ | 指出冲突、隐含取舍和阻塞条件 |
| 相似实现 | 仓库里有哪些先例，哪些能复用、哪些必须隔离？ | 对照真实路径、差异、公共契约、测试和历史踩坑 |

雷达对外摘要至少说明：当前最重要的结论、已确认事实、最重要未知项、需要用户决定的取舍、是否可以进入需求基线确认。详细证据和稳定 ID 写入产物，不要求用户理解机器记录结构。

雷达按风险自适应：低风险可以很短，高风险或跨边界需求扩大搜索。能在仓库内找到类比时必须先做真实对照；没有找到时记录搜索范围和查询依据，不把“没想到”当作不存在。

若某个未知事实需要独立调查，先说明调查问题、理由、角色、预期产物和是否阻塞基线，再由用户选择调查 Agent；Harness 不自动选择或启动调查 Agent。阻塞需求基线的 `RAD-xxx` 未清零时不得批准需求基线。

这五个问题是一次性给出的汇报框架，不是发给用户的问卷：雷达结论连同证据整表一次汇报；任何需要用户决定的取舍不在汇报中直接提问，按[用户决策协议](#用户决策协议)逐个提出。

## 文档锚定的 Grill

需求基线是“用户决策协议”最重的应用场景：自适应需求雷达先查清事实并沉淀问题队列，凡属取舍、风险容忍、范围或无法安全推导的冲突，按[用户决策协议](#用户决策协议)实际调用 `grilling`，一次只问一个需要用户决定的问题，并给出推荐答案。能从仓库、现有文档或工具查到的事实先查，不拿事实问题消耗用户。

同时实际调用 `domain-modeling`：

- 术语模糊、重载或与已有语言冲突时立即提出规范术语；
- 用具体边界场景检验领域关系；
- 每次术语达成一致，立即更新 `核心文档/领域词汇.md`；
- 对难以逆转、缺少上下文会令人困惑、且确有取舍的决定，写入 `核心文档/决策账本.yaml`；不要把实现细节塞进词汇表。

每个回答同步维护需求雷达、问题与发现、问答摘要及稳定 ID，并写入 `过程产物/需求审问/用户确认记录.md`。每条记录必须包含「问题、用户答复原文、结果 DEC/RAD ID」，不得把 Agent 的转述当作用户原话。整个需求基线阶段无需向用户提问时，也必须在该文件记录「无需提问的理由」以及所依据的仓库或既有决策证据。

Grill 完成的条件：目标、非目标、用户/场景、约束、失败与边界、验收信号、未决问题都明确，且没有阻塞路线比较的歧义。需求基线门禁的批准话术必须引用当前 `用户确认记录.md` 的绝对路径或产物版本，说明用户正在确认哪些原始答复与无需提问理由；不能只引用聊天摘要或单独的 `DEC-*`。

汇总 `核心文档/需求基线.md`，明确列出仍未知但可在后续验证的事实。将状态设为 `needs_user`，等待用户对具体版本通过 `requirements_baseline`。未确认时不得进入路线深化。

## 路线卡与用户选择

先判断是否存在真正的高代价分歧：模块 seam、状态所有权、数据一致性、迁移策略、外部依赖、兼容性或交付路线。没有实质分歧时生成一张推荐卡；有分歧时生成二至三张。

路线卡只包含核心思路、改动边界、关键决策组合、优点、代价、最大风险、适用条件、验证需要和推荐程度。每张保持简洁，不生成多份完整技术规格。

涉及模块设计时实际调用 `codebase-design`，用以下问题比较路线：接口是否小、实现是否隐藏复杂度、seam 是否真实、调用者获得多少 leverage、变更是否集中、接口是否也是自然测试面。

把路线卡写入 `过程产物/路线评估/`，向用户展示差异和推荐，随后停止并等待 `route_selection`。记录选择、否决路线和原因到 `DEC-xxx`。用户未选时不得继续。

推荐是建议而不是授权。用户明确选择非推荐路线时尊重用户选择，记录理由和风险，不得自动改回推荐路线；只有发现新事实使路线不再可实施时才登记 `route_failure` 并转交 `grh-recover`。

## 研究或原型

仅当路线或规格依赖未验证事实时进入此阶段：

- **研究**：答案存在于权威外部资料、协议、发行说明、许可证或供应商文档；优先使用已安装 `research`，缺失时生成研究角色任务包。
- **原型**：需要观察 UI、状态模型、性能轮廓、库行为或技术可行性；优先使用已安装 `prototype`，缺失时生成一次性原型任务包。
- 两者都需要时，先研究会改变原型边界的事实，再做最小原型。

所有任务、原始证据和结论写入 `过程产物/研究与原型/`。原型必须有待回答问题、时间盒、成功/失败判据和删除/隔离策略；原型代码不是生产实现，也不自动进入产品仓库。验证结论更新风险、决策和所选路线，不重开已确认需求访谈。

若不需要研究或原型，必须在所选路线评估或最终规格中记录一句「跳过研究/原型」的理由，说明哪些事实已经足够、为何不会削弱路线或规格可信度。

## 多仓双基线与批准复用

多仓项目同时维护两种不同用途的基线，不能混为一个哈希：

- **工作流 owner 基线（`owner baseline`）**：绑定工作流所属仓库，通常是聚合根；状态迁移、证据当前性和工作流恢复以它为准。
- **目标子仓基线（`target_repo_baseline`）**：记录真正承载产品改动的目标子仓；用于判断产品实现、路线与任务证据是否漂移。

聚合根出现与当前需求无关的脏改动，不等于目标子仓路线已经失效。Agent 必须分别说明两条基线发生了什么，不得用“baseline 变了”掩盖差异，也不得清理、提交或恢复用户的 owner worktree 文件。

同一工作流内可以复用用户此前批准的 `ignore_owner_worktree_changes` 策略，但每次必须登记新的 CHG 和证据。新 CHG 自身必须声明 `status: approved`、`operation: baseline_reconcile`、当前 `workflow_id`、`ignore_owner_worktree_changes: true`，并以 `related_change` 指向同一工作流内 `status: approved`、`approved_by: user`、`operation: baseline_reconcile`、`ignore_owner_worktree_changes: true` 的旧 CHG；同时显式声明 `route_invalidated: false`、`product_change: false`。新记录的批准来源是 `related_change`，没有本轮新答复时不得伪造新的 `approved_by: user`。任一条件不满足就停止并重新请求用户决定。复用只继承这项 owner baseline 处置策略，不继承路线选择、产品变更授权或任何门禁批准。

推荐对用户这样解释：“目标子仓及已选路线没有变化；变化只发生在工作流所属的聚合根。你先前已经批准忽略这类 owner worktree 改动，本次会用新的变更记录关联那次批准，并刷新当前证据；不会修改、清理或提交聚合根中的其他文件。”

## 仓库挑战与最终规格

为独立调查者生成 `过程产物/仓库挑战/` 任务。挑战者必须读取真实仓库、完整相关文件和调用路径，逐项验证所选路线的假设：入口与调用者、同步/异步路径、数据与事务、并发、错误、配置、安全、迁移、复用、测试和兼容性。

对模块或接口假设实际调用 `codebase-design`。挑战报告对每项假设给出成立、部分成立、不成立、遗漏、需用户决策或无法验证，并附路径、符号和证据。主会话逐项核验，不盲信挑战者总结。

只深化已选路线，形成唯一 `最终产物/最终规格.md`。规格包含：目标/非目标、需求与决策映射、真实仓库事实、模块与接口、数据/状态/错误/兼容/迁移、安全与运维、实施边界、测试策略、验收标准、风险和回滚。未验证假设必须显式标注。

最终规格必须包含外部依赖表，逐项记录外部接口、协议、供应商能力或运行合同的验证证据、风险和成熟度：`verified` 表示已有当前权威证据；`provisional` 表示暂按明确假设继续；`blocked` 表示缺失事实已经阻止安全实施。每个 `provisional` 或 `blocked` 项都必须登记关联 `RAD-*`，并明确写入 `blocking_level: implementation`；这些雷达项未解决前不得进入实施。

把最终规格置为 `needs_user`，等待用户批准具体版本的 `final_spec_approval`。门禁话术必须逐条点名全部未验证外部合同、对应 `provisional`/`blocked` 状态及其对实现、兼容性和验收的具体风险，不得用笼统的“仍有外部风险”替代。批准前产品仓库仍只读。`grh-plan` 在这里停止等待；最终规格批准后生成执行 Frontier 并停止，同时只生成任务图和任务包，建议显式调用 `grh-run`，不得自行开始实施。

## 垂直切片与安全并行

最终规格批准后，按用户可观察能力或可独立验证的 tracer bullet 拆分任务。每个切片应尽可能贯穿必要层次并产生真实可验证结果；不要默认按“前端/后端/数据库”横向分层。

每个任务必须声明：目标、稳定 ID、依赖、输入版本、允许/禁止范围、精确写路径、共享契约、迁移/生成物、Git 基线、worktree/分支、测试与验收、停止条件、任务包和报告路径。

任务图多于一个切片时，必须生成 DAG、明确的依赖/阻塞边、当前 Frontier（当前可开工集合）和并行组；不能只列任务顺序。单个任务覆盖全部 `REQ-*` 时，必须在任务图的 `non_split_reason` 记录「不拆的理由」，说明为何它仍能在一个新会话内独立验证，不能用“任务较小”替代证据。

仅当以下全部成立才可并行：

- 无依赖或阻塞关系；
- 写路径无重叠，且不会由生成器间接重叠；
- 不修改同一公共接口、schema、锁文件、配置、迁移、协议或共享契约；
- 使用不同 worktree/分支；
- 集成顺序和验收边界明确。

任何信息缺失即判定不安全，改为串行。并行组完成后必须做集成检查，不能只看各任务单独通过。

## 角色执行循环

按 `角色任务协议.md` 为实施、审查、修复、集成和验收分别生成本地任务包与短启动提示词。用户选择模型和 Agent，在新本地会话启动；Harness 不自动替用户选模型。

实施按任务性质优先使用已安装的 `tdd`：先约定可观察的测试 seam，再以一条失败测试和最小实现逐切片推进；不适用或跳过时在实施任务中记录不使用理由。`implement` 或 `tdd` 缺失时仍执行 GRH 实施任务包和证据契约，不阻塞工作流。

审查优先参考已安装 `code-review` 的双轴纪律，Standards 与 Spec 分开给结论。每条发现必须形成结构化 comments，至少包含 `axis: standards|spec`、分类（blocking / must_fix / optional / invalid）、路径或符号证据、关联 `ISSUE/EVD` 和处置状态，并经 `task-review` 追加登记进任务 `review_history`；只写散文审查报告、保持 `comments: []` 不算完成审查。

循环为：实施报告 → 独立审查 → 问题成立性核验 → 最小修复 → 针对性复审。全部任务结束后进行并行结果集成检查，再由新的独立会话最终验收。每一步只在证据有效且状态脚本允许时推进。

进入 `independent_assurance` 前逐条对账需求雷达：所有仍为 open 且 `blocking_level: implementation` 的 `RAD-*` 必须变为 `resolved`、`superseded` 或 `dismissed`；`superseded` / `dismissed` 必须带替代项或理由。不得把实现已完成当作雷达项自动关闭。

`grh-run` 只处理任务包、实施和经批准的修复循环，不执行最终验收。`grh-check` 只读审查、集成检查和验收；发现实现问题时记录审查历史并建议修复任务，不得修改产品代码或自动调用 `grh-run`。
