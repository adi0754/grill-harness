# 文档与产物契约

## 唯一写入边界

所有 Grill Harness 状态、提示词、任务包、报告和知识文档只能写入当前工作流目录：

```text
~/.grill-harness/项目/<项目简称>-<项目短ID>/工作流/<日期>-<需求简称>-<工作流短ID>/
```

除已通过最终规格门禁且任务包明确授权的产品实现外，目标仓库只读。禁止把 Harness 指针、`.grill-harness`、需求文档、路线卡、任务包或报告写进产品仓库。`~/.codex`、`~/.claude` 等 Agent 目录也不是运行时产物目录。

## 当前工作流结构

```text
核心文档/
  需求基线.md
  决策账本.yaml
  领域词汇.md
  当前规格.md
  任务图.yaml
过程产物/
  需求审问/
    需求雷达.md
    问题与发现.yaml
    用户确认记录.md
    相似实现对照.md
  路线评估/
  研究与原型/
  仓库挑战/
  任务交接/
  实施报告/
  审查修复/
  学习草稿/
最终产物/
  最终规格.md
  集成报告.md
  验收报告.md
  项目经验.md
系统/
  state.yaml
  artifacts.yaml
  tasks.yaml
  evidence.yaml
  failures.yaml
```

- **核心文档**：随当前决策维护的权威工作合同；旧版本通过系统索引保留。
- **过程产物**：可审计的访谈、候选、调查、任务包和角色报告，不代表最终结论。
- **最终产物**：用户批准的唯一规格、集成和验收结论、可复用经验。
- **系统**：机器协议与索引，不覆盖人读产物，也不代替真实仓库事实。

模板来自 Skill 的 `assets/templates/`，实例必须复制到上述对应目录。模板中的输出路径一律改为当前工作流下的绝对路径。

需求发现的固定实例路径是 `过程产物/需求审问/需求雷达.md`、`过程产物/需求审问/问题与发现.yaml`、`过程产物/需求审问/用户确认记录.md` 和 `过程产物/需求审问/相似实现对照.md`。知识草稿只写 `过程产物/学习草稿/`；失败追加 manifest 为 `系统/failures.yaml`。

## 权威顺序与稳定 ID

真实仓库说明当前实现事实；决策账本说明用户确认的意图；已批准最终规格说明实施合同；任务图与任务包说明局部授权；证据说明是否实现；系统文件只说明索引和流程位置。冲突时不按文件新旧或内容完整度自动选边。

稳定 ID：`RAD-001`、`REQ-001`、`DEC-001`、`CON-001`、`RISK-001`、`CHG-001`、`TASK-001`、`ISSUE-001`、`EVD-001`、`KNW-001`。每条记录包含版本、状态、来源、依赖/覆盖关系和替代项。修改增加版本并保留旧版本，禁止静默覆盖。

多仓工作流同时记录工作流 owner 基线与 `target_repo_baseline`。前者绑定聚合根状态和工作流证据当前性，后者绑定真正承载产品改动的目标子仓；任何报告都必须分别说明两者，不得用其中一个替代另一个。复用已批准的 owner baseline 忽略策略时，新 CHG、`related_change`、新证据和 baseline reconciliation 记录都必须追加保留。

## 产物最低契约

| 产物 | 必需内容 |
|---|---|
| 需求基线 | 目标、非目标、场景、需求/约束、边界、验收信号、未决项、门禁版本 |
| 需求雷达 | 五类雷达信号、`RAD-xxx`、阻塞级别、证据、处置状态和需求基线映射 |
| 用户确认记录 | 逐条保存全流程需要用户决定的问题、所属阶段或门禁、用户答复原文、结果 `DEC/RAD` ID；未提问的阶段或门禁保存无需提问理由和事实依据 |
| 相似实现对照 | 搜索范围、路径/符号、相同与差异、可复用点/测试、公共契约和新增行为 |
| 领域词汇 | 规范术语、定义、反例/易混词、领域关系；不含实现细节 |
| 路线卡 | 核心思路、边界、决策组合、优劣、风险、验证需要、推荐度 |
| 研究/原型 | 问题、方法、原始证据、结论、限制、对决策的影响 |
| 仓库挑战 | 被挑战假设、真实路径/符号/调用链、证据、结论、规格影响 |
| 最终规格 | ID 映射、仓库事实、接口/数据/错误/兼容/迁移、测试、验收、风险 |
| 任务图 | DAG、阻塞边、Frontier、写路径、共享契约、并行判定、Git 基线；单任务覆盖全部需求时包含 `non_split_reason` |
| 角色任务包 | 首段明白中文 `给用户的话`、角色目标、授权/禁止范围、停止条件，以及集中在 `证据与标识` 的绝对路径、输入版本和输出契约 |
| 角色报告 | 实际读取/修改、完整 diff、命令证据、ID 覆盖、问题和剩余风险 |
| 验收报告 | 对当前基线的逐条验收、证据、中文主结论（通过 / 有条件通过 / 不通过 / 无法验证）和发布阻塞项；机器码仅作附注 |
| 失败清单 | `系统/failures.yaml` 的追加式失败链、四类失败、指纹、尝试次数、前序哈希和恢复决定 |
| 审查历史 | 任务 `review_history` 的追加式记录、基线、评论分类、处置与前序哈希；当前 review 必须由历史头派生 |

## 证据与长期知识

证据记录实际命令、绝对工作目录、开始/结束时间、退出码、Git 基线、产生者、原始输出绝对路径和可复现性。摘要可以链接证据，不能替代证据。证据与当前基线不符即失效。

### 证据新鲜度分级

- **基线绑定证据**：测试、构建、静态检查及其他由当前仓库内容决定的命令证据，`expires_at` 默认取 `validated_at` 后 30 天。它的失效主因是基线漂移而非时间；即使未到期，只要工作流 owner 基线或相关 `target_repo_baseline` 改变，也必须立即失效并在新基线上重跑原命令。
- **外部时效证据**：第三方接口可用性、供应商状态、线上配置或其他会独立于仓库变化的外部事实，`expires_at` 默认取 `validated_at` 后 24 小时。记录必须注明可复现的复验命令；到期或外部条件变化后执行该命令并登记新的 `EVD-*`，不得只延长旧记录时间。

模板中的 TTL 只用于计算最终绝对 `expires_at`，正式证据仍必须写完整时间戳、证据分级和实际复验命令。若有更短的权威有效期，采用更短期限并记录来源。

知识分三种运行方式：

1. `knowledge-query` 只读查询 `知识库/项目知识/<项目ID>/knowledge.yaml`，可选同时查询 `知识库/通用知识/knowledge.yaml`；只有 `verified` 记录可指导规划。
2. `knowledge-draft` 把 `tentative` 记录写入当前工作流的 `过程产物/学习草稿/`，不改变长期知识。
3. `knowledge-promote` 先生成知识变更预览，用户以绑定预览版本的 `DEC/CHG` 批准后才能写项目知识；提升到通用知识还需要第二次独立批准，并且项目知识归档已经完成。

正式归档要求当前验收结论被接受、前序必需阶段完成、Git 基线与证据仍有效且至少有一条已批准提升记录。未验收工作流不能伪装成完成；确认的 `route_failure` 只允许按独立失败例外沉淀项目级知识，不能完成归档或直接进入通用知识。

长期文档不保存完整聊天、临时推理、凭据、环境变量值或整段日志。项目经验只保留稳定决策、否决路线及原因、领域知识、重要仓库事实、踩坑、证据索引、剩余风险和下次建议。
