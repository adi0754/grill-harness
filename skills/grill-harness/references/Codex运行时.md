# Codex 运行时

在新的本地 Codex 任务中发送 `角色任务协议.md` 生成的短启动提示词。任务包路径、产品项目目录和报告路径都必须是绝对路径；Codex 从任务包和真实仓库重建上下文，不依赖原任务历史。

允许使用任务所需且已安装的 Skills。必需能力缺失时停止并报告，不静默模拟或安装。若使用子 Agent，任务报告要记录角色、输入隔离和独立性；子 Agent 不自动等于不同模型验证。

创建/切换分支或 worktree、合并、提交、推送、发布及破坏性 Git 操作必须在任务包中明确授权，否则停止。正式角色只修改授权产品路径和指定的 `~/.grill-harness/` 报告路径，不能直接批准门禁或状态。

## Harness 状态与 Codex 会话 Goal 映射

| Harness 状态 | 含义 | Codex 会话 Goal 呈现 | Codex 必须动作 |
|---|---|---|---|
| `needs_user` | 正在等待用户对真实门禁产物作决策 | `blocked-on-user` | 停止后续阶段，复述待决问题、选项、推荐项、待绑定产物版本和预分配 DEC ID；收到用户原文前不得假装继续 |
| `blocked` | 已有证据表明外部依赖、权限、输入或工作流冲突阻止当前动作 | `blocked` | 复述具体阻塞条件、已有证据、已尝试动作和解除条件；仍可安全调查时只做调查，不扩大授权范围 |

`blocked-on-user` 是面向用户的等待语义，用来区分人工门禁与一般故障；Codex 侧仍按当前运行时允许的 Goal 状态字段记录，不得反向改写 Harness 的磁盘事实。用户答复后先落档原文和对应 `DEC/CHG`，再重新读取状态决定是否解除等待。

完成时返回报告绝对路径、当前 Git 基线和验证结论。没有真实命令证据时使用“无法验证”，不能声称完成。
