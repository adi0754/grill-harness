# 测试、审查、修复、集成与验收

## 证据先于结论

每项实施证据必须包含实际命令、绝对工作目录、时间、退出码、Git 基线、产生者、原始输出绝对路径、可复现性以及对应 `REQ/DEC/TASK/ISSUE`。检查完整 diff、未提交/未跟踪文件和越权范围。命令未运行、输出被截断、基线变化或只提供口头摘要时，不得声称通过。

优先运行仓库已定义的测试、类型检查、lint、构建和集成命令。无法运行时写明原因、已验证的较小范围和剩余风险，不把“无法验证”改写成“应该没问题”。

## 独立代码审查

审查任务默认只读，并行检查两条轴：

- **Standards**：正确性、安全、维护性、性能、错误处理、并发/事务、测试质量、仓库规范、意外文件和敏感信息。
- **Spec**：`REQ/DEC/CON` 覆盖、非目标、允许范围、接口/数据/迁移/兼容性、验收标准和未验证假设。

每个发现使用稳定 `ISSUE-xxx`，记录严重度、路径与符号、复现/推理证据、影响、规格映射和建议方向。审查者必须读完整相关文件和真实 diff；上一位 Agent 的实施摘要只作为线索。

无重要问题不等于完成：还需证据有效、任务验收通过，并完成后续集成和最终验收。

## 问题修复与复审

修复任务先验证每个 `ISSUE-xxx` 是否成立：成立、部分成立、不成立或无法验证。对成立问题作能恢复规格符合性的最小修改，保留原问题与处置关系。

需要改变已批准需求、共享契约、迁移策略或任务边界时停止，创建 `CHG-xxx`，不得借修复扩大范围。修复后运行针对性测试及受影响回归测试，并由独立审查者复审原问题；修复者不能自关重要问题。

每次审查通过 `task-review` 追加到任务 `review_history`，不得覆盖旧结论。阻塞项和 must-fix 项必须在当前 Git 基线上被后续独立复审关闭；无效评论保留理由。仅剩可选建议时可记录其 `ISSUE-xxx` 并收敛，不能把可选建议偷偷升级为当前交付范围。

## 失败分类与恢复

所有失败先通过 `failure-record` 写入 `系统/failures.yaml` 的追加式 manifest，并用稳定问题、失败验收/命令和 Git 基线形成指纹。四类失败及其边界为：

- `implementation_failure`：路线仍成立但实现或修复未满足已批准规格；同一失败链第一次和第二次允许普通修复，第三次阻断普通修复并要求 `grh-recover` 与真实用户批准。
- `route_failure`：路线假设失效；停止实施，由 `grh-recover` 生成带新事实的路线卡，等待用户重新选择路线，不能自动换路。
- `evidence_failure`：命令、输出、基线或可复现证据不足；回到补证据，不能把缺证据改写成实现失败或通过。
- `workflow_integrity_failure`：状态、manifest、review history、门禁或产物链不一致；先 reconcile，不能继续业务修复。

阈值覆盖、新失败链、恢复模式和路线重选都必须绑定真实 `DEC/CHG` 批准。失败类和追加历史不能在同一指纹链中被静默改写。

## 并行结果集成检查

一个并行组完成后，集成者在共同目标基线上检查：

- 分支/worktree 与提交基线、合并冲突和遗留未提交文件；
- 公共接口、schema、锁文件、配置、迁移顺序、生成物和重复实现；
- 各任务的假设是否互相兼容，端到端数据与错误路径是否闭合；
- 全量构建、测试、类型检查、lint 和必要的集成/迁移测试；
- 最终 diff 是否仍覆盖已批准规格且没有越权。

集成修复必须在任务包明确授权范围内；发现共享契约冲突或产品决策时返回用户。只有单任务时也要做简化集成检查，确认最终仓库状态而不是只看实施报告。

## 最终独立验收

使用新的本地 Agent 会话，从 `核心文档/需求基线.md`、`最终产物/最终规格.md` 和真实仓库重新理解目标。验收者不得依赖原聊天，不修改产品代码或批准记录，并必须实际运行可执行命令。

逐项建立 `REQ/DEC/CON → 实现路径/行为 → EVD` 映射，核对最终 Git 基线、完整 diff、未提交状态、集成报告、已关闭问题和剩余风险。结论只能是：

- **验收通过**：当前基线满足全部批准项，证据有效，无发布阻塞项。
- **有条件通过**：核心目标满足，但存在明确、被用户接受且不阻塞当前交付的条件。
- **验收不通过**：至少一个批准项不满足或存在阻塞问题。
- **无法验证**：环境、输入、权限或证据不足，不能形成可靠结论。

只有“验收通过”允许主会话宣称完成。是否合并、发布或部署仍是单独的用户决定。

## 知识沉淀与归档

验收后先写入 `最终产物/项目经验.md` 和 `过程产物/学习草稿/`：目标与结果、选中/否决路线、关键决策、领域语言变化、重要仓库事实、实施偏差、问题与踩坑、证据索引、剩余风险和下次建议。学习草稿仍是 tentative。

正式知识采用两步归档：先生成变更预览并经用户批准提升到项目知识；只有项目归档完成后，候选通用知识才可通过第二次独立批准写入通用知识。未被接受的验收、缺少当前证据或未批准预览都必须拒绝正式归档。

不保存完整聊天、临时推理、凭据或大段日志。失败或无法验证的工作流也要归档真实结论，不能伪装成完成。
