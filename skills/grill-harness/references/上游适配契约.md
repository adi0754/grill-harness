# 上游适配契约

## 领域语言

- **能力（Capability）**：Grill Harness 会调用并依赖其行为契约的 Skill。`grilling`、`domain-modeling`、`codebase-design` 是必需能力。
- **兼容参考（Compatibility Reference）**：只用于比较工作流设计，不是可调用依赖。`grill-with-docs` 始终属于此类。
- **预检（Preflight）**：先读取 `npx skills list --json`，再用已知 Skill 根目录、`SKILL.md` 元数据和符号链接目标验证所得清单。
- **固定清单（Pinned Manifest）**：一次检查所观察到的上游事实快照，不表示接受或安装该版本。
- **行为契约（Behavior Contract）**：Harness 实际依赖的可观察行为；其变化比纯内容哈希变化风险更高。

## 预检接口

预检是只读深模块。它返回必需、可选和兼容参考项的状态、验证路径、安装或更新建议，以及是否允许进入依赖阶段。CLI JSON 是首要发现来源；文件系统和元数据是验证来源。CLI 不存在时可用文件系统降级，但不得猜测历史命令语法。

安装和更新命令只能从当前 CLI 的帮助输出生成。输出是交给用户审核的建议；预检不得执行安装、更新、覆盖或修复动作。缺少必需能力时必须失败关闭；缺少可选能力或兼容参考只记录降级。

## 固定清单字段

清单必须完整记录：`repository`、`ref`、`commit`、`checked_at`、`upstream_updated_at`、`license`、`source_paths`、`hashes`、`behavior_contracts`、`local_differences`、`risks`、`last_test_results`。

## 兼容性分类

- `metadata-change`：ref、许可证或时间等元数据变化。
- `added` / `removed` / `renamed`：能力源新增、删除或重命名；删除和重命名需要人工决策。
- `content-fix`：路径和行为契约不变，但内容哈希变化。
- `content-change`：内容哈希与路径或行为契约同时变化，必须与其它分类共同报告。
- `behavior-contract-change`：Harness 使用的可观察行为发生变化；建议暂缓更新。

离线检查只比较本地可得事实，不触发网络访问。在线检查也只读取远端事实。所有报告必须明确 `actions_performed: false` 和 `accepted_upstream_changes: false`；任何安装、更新、覆盖或接受都必须由用户另行批准并在适配流程之外执行。
