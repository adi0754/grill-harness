# 角色任务协议

## 目标

用本地自包含任务包把实施、审查、修复、集成和验收隔离到新的 Agent 会话。用户决定使用什么模型和 Agent；Grill Harness 不替用户选择，也不要求多个角色使用同一厂商。

任务包是角色的完整工作合同，但不得附带原始聊天历史。角色必须从已批准产物和真实仓库重建上下文，不能只相信上一角色的摘要。

## 落盘位置

所有文件都在当前工作流目录：

- 任务包和启动提示词：`过程产物/任务交接/<TASK-ID>-<角色>-任务.md`、`<TASK-ID>-启动提示词.md`
- 实施报告：`过程产物/实施报告/<TASK-ID>.md`
- 审查、修复和复审报告：`过程产物/审查修复/<TASK-ID>-<类型>.md`
- 集成报告：`最终产物/集成报告.md`
- 验收报告：`最终产物/验收报告.md`

目标产品仓库不得保存任务包或报告。

## 每份任务包的固定字段

1. 首段 `给用户的话`：使用明白中文标题和不超过五句的 30 秒结论，先解释目标、影响、停止点和当前不能宣称什么。
2. 角色、单一目标、已确认仓库事实、允许/禁止范围、共享契约和并行组。
3. 要执行的步骤、验收标准和停止条件：输入缺失、基线漂移、规格冲突、越权范围、新产品决策、危险操作或无法验证。
4. `证据与标识`：对应 `TASK/REQ/DEC/CON/ISSUE`，产品与工作流绝对路径、报告绝对输出路径、Git 基线、分支/worktree、输入版本、测试/检查命令和证据格式。
5. 输出结构：事实、动作、完整 diff、证据、ID 覆盖、问题、剩余风险和明确结论。

验收任务先用中文给出“通过 / 有条件通过 / 不通过 / 无法验证”，再在 `证据与标识` 中附 `PASS / PASS_WITH_LIMITATION / FAIL / UNVERIFIED` 等机器码；机器码不得替代对用户的理由和限制说明。

只读角色的允许修改范围仅包含任务包指定的 Harness 报告绝对路径。产品代码保持只读。

## 短启动提示词

为每个任务包单独生成本地启动提示词，只做定位和纪律约束，不复制任务正文：

> 使用 `$grill-harness` 执行角色任务。读取任务包 `<任务包绝对路径>`，在 `<项目绝对路径>` 检查真实仓库和 Git 基线，只执行任务包授权范围，运行要求的验证，并把完整报告写入任务包指定的 `~/.grill-harness/` 绝对路径。不要依赖原聊天或上一位 Agent 的总结；遇到停止条件立即停止并报告，不自行扩大范围或批准门禁。

新会话必须能访问本地产品仓库、任务包和输出目录。第一版不生成网页模型便携提示词，不要求用户粘贴完整上下文。

## 角色隔离

- **实施者**：只完成一个垂直任务，不批准自己的实现。
- **审查者**：默认只读；同时检查 Standards 和 Spec，读取完整文件、真实 diff、未提交状态和证据。
- **修复者**：先验证 `ISSUE-xxx` 是否成立，再作最小修复；新范围退回用户。
- **集成者**：检查任务间契约、迁移、配置、生成物和端到端行为，不新增产品范围。
- **验收者**：新的独立本地会话，从需求基线与最终规格重新理解目标，只读验收当前基线。

同一个 Agent 会话不得同时承担实施和最终验收。子 Agent 可以提供并行调查，但只有在任务包明确且独立性有记录时才算正式角色隔离；“调用了子 Agent”不等于“使用了不同模型”。

## 调度与回收

主会话只派发当前 Frontier 中、状态合法且并行安全的任务。角色必须按指定绝对路径写回报告。每份报告返回后，主会话亲自检查输出文件、仓库状态、完整 diff 和证据，再更新状态；角色不得直接把门禁或工作流标成通过。

报告缺失、路径错误、基线不符、只给摘要或证据不可复现时，任务仍未完成。修复后必须由独立审查者复审相关问题；并行组全部完成后必须进入集成检查。
