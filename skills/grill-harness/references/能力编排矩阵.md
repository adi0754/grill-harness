# 能力编排矩阵

本矩阵把上游 Skill 的关键纪律内化为 Grill Harness 的协议与产物契约。运行时可以调用已安装的上游 Skill，但磁盘中的 GRH 产物始终是可审计事实；可选 Skill 缺失时使用表中降级产物，不静默假装调用成功。

必需能力仍仅为 `grilling`、`domain-modeling`、`codebase-design`。除这三项外，其余能力全部可选，缺失不阻塞工作流，也不会被自动安装或更新。`grilling` 缺失时按必需能力规则失败关闭，不使用降级产物绕过需求基线门禁。

| 意图 | 上游 Skill 与运行时地位 | 内化纪律要点 | 缺失时的 GRH 降级产物 | 验证锚点 |
|---|---|---|---|---|
| 不理解 / 需人确认 | `grilling`（必需能力之一） | 仓库可查的事实先查；只把决策交给用户；一次只问一个问题并给推荐答案；达成共同理解前不实施 | 无静默降级；失败关闭。能力可用时将逐问结果写入 `过程产物/需求审问/用户确认记录.md` | [阶段执行协议：文档锚定的 Grill](阶段执行协议.md#文档锚定的-grill)、`assets/templates/用户确认记录.md` |
| 巨大模糊工作 | `wayfinder`（可选）或现有 Wayfinding 模式 | 先命名目的地；地图只做索引；把当前可回答的问题做成独立调查/决策任务；维护阻塞边、当前 Frontier 与尚未清晰的“雾区”；默认只找路、不提前交付终点 | Wayfinding 模式下的需求调查任务包、`问题与发现.yaml`、路线评估产物和后续任务图 | [阶段执行协议：预检与对齐](阶段执行协议.md#预检与对齐)、`assets/templates/需求调查任务.md` |
| 对齐后定规格 | `to-spec`（可选） | 基于已确认对话与真实仓库综合，不重复发起需求访谈；说明问题、用户结果、实施与测试决策、非目标；优先复用最高且最少的测试 seam；只深化已选路线 | 唯一 `最终产物/最终规格.md`，由 `规格.md` 模板承载需求/决策映射、测试 seam、风险和批准门禁 | [阶段执行协议：仓库挑战与最终规格](阶段执行协议.md#仓库挑战与最终规格)、`assets/templates/规格.md` |
| 拆任务 | `to-tickets`（可选） | 按可独立验证的 tracer bullet 做垂直切片；逐项声明真实阻塞边和当前可开工集合；宽机械改造使用 expand–migrate–contract；单任务覆盖全部需求时写明不拆理由 | `核心文档/任务图.yaml`、本地角色任务包、阻塞边和 Frontier | [阶段执行协议：垂直切片与安全并行](阶段执行协议.md#垂直切片与安全并行)、`assets/templates/任务图.yaml` |
| 逐切片实施 | `implement`（可选），可叠加 `tdd`（可选） | 一次只完成一个已授权切片；在约定测试 seam 上按 red → green 逐条推进；定向测试常跑、收尾跑全量；检查完整 diff；不得因上游建议绕过任务包或 Git 约束 | `实施任务.md` + `过程产物/实施报告/<TASK-ID>.md` + 当前基线命令证据；未使用 TDD 时记录理由 | [阶段执行协议：角色执行循环](阶段执行协议.md#角色执行循环)、`assets/templates/实施任务.md` |
| 审查 | `code-review`（可选） | 固定审查基线和真实完整 diff；Standards 与 Spec 两轴独立检查，不互相掩盖；发现以结构化 comments 记录分类、证据和处置，不以散文报告代替状态事实 | `审查任务.md` + 审查报告，并通过 `task-review` 追加进任务 `review_history` | [阶段执行协议：角色执行循环](阶段执行协议.md#角色执行循环)、`assets/templates/审查任务.md` |

更完整的阶段动作、门禁和停止边界见 [阶段执行协议.md](阶段执行协议.md)。该协议反向引用本矩阵，避免把能力名称、运行时依赖和 GRH 落盘契约分散成彼此冲突的说明。
