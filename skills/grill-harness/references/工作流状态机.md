# 工作流状态机

本文件解释如何根据机器状态路由。阶段合法迁移、门禁校验和产物一致性以 `scripts/state.py`、`status` 与 `reconcile` 的 JSON 结果为权威。

## 三种运行模式

| 模式 | 使用条件 | 保留内容 |
|---|---|---|
| 轻量 | 局部、低风险、单任务可闭环 | 只读侦察、精简产物、三个门禁仍分别批准、真实 diff/测试、独立验收 |
| 标准 | 默认；有设计取舍或跨模块改动 | 三个门禁、路线卡、仓库挑战、垂直切片、角色隔离、集成与验收 |
| Wayfinding | 巨大、模糊、单会话无法建立可信地图 | 独立调查/决策任务包；每次解决一个问题，收敛后回到标准模式 |

模式不是质量等级。轻量模式不能跳过用户授权和证据；Wayfinding 不能在方向未清楚时实施。

Wayfinding 模式中，每个新会话只解决一个调查或决策问题；调查结果是标准流程的输入，不是另一套实施流程。

## 执行顺序与准入

下列顺序与 `state.py` 的十二个机器阶段一一对应。自适应需求雷达是 `requirements_baseline` 阶段内形成基线前的必需活动；研究/原型是 `design` 或 `repository_challenge` 内按问题选择的可选产物；两步知识提升是 `knowledge_archive` 阶段内的归档协议。它们都不新增机器阶段。

1. **预检**：项目身份明确，必需能力可验证。
2. **对齐**：运行模式、目标、成果、范围和风险容忍明确。
3. **需求基线**：先完成自适应需求雷达，再用文档锚定 Grill 与领域语言收敛；等待用户确认。
4. **设计**：识别真正路线分歧，使用 `codebase-design` 比较模块形状；需要时产生研究/原型产物。
5. **路线选择**：只展示一至三张路线卡；等待用户选择。
6. **仓库挑战**：独立核验高层方案与真实仓库的差异；需要时产生补充研究/原型产物。
7. **规格**：只深化选中路线，形成唯一最终规格。
8. **最终规格批准**：等待用户批准具体规格版本。
9. **任务拆分**：垂直切片、DAG、Frontier、冲突和并行组。
10. **实施**：独立新会话按任务包执行，随后审查与修复。
11. **独立保障**：并行结果集成检查与最终独立验收。
12. **知识归档**：先形成学习草稿，再经用户批准提升到项目知识；通用知识需要第二次明确批准。

研究/原型产物不得标记为机器 phase，也不得把所属必需 phase 标记为 `skipped`。不需要研究或原型时只在阶段产物中记录“不产生该可选产物”的理由，机器阶段仍按正常状态推进。

详细阶段动作读取 `阶段执行协议.md`。缺少前置产物或门禁时不得静默跳阶段。

## 三个人工硬门禁

三个门禁分别回答不同问题，不得统一描述成“用户确认是否继续”：

| 门禁 | 用户真正回答的问题 | 绑定产物 | 通过后允许 | 防止的问题 |
|---|---|---|---|---|
| `requirements_baseline` | 我们理解的是不是同一个需求？ | `核心文档/需求基线.md` 的具体版本 | 生成与比较路线 | 带着错误目标、遗漏边界或阻塞歧义设计方案 |
| `route_selection` | 这些可行路线中选择哪一条？ | 路线卡与 `DEC-xxx` | 研究/原型、仓库挑战、深化所选路线 | Agent 默认替用户决定成本、范围和风险取舍 |
| `final_spec_approval` | 这份实施合同是否足以授权写代码？ | `最终产物/最终规格.md` 的具体版本 | 拆分、建执行分支/worktree、实施 | 把草案、推荐或未验证假设当成实施授权 |

批准必须记录用户决定、时间、产物版本和 `DEC-xxx` 或 `CHG-xxx`。不能把“继续”“看起来可以”或旧会话摘要解释为批准。

轻量模式只缩短雷达、路线卡和规格，不合并授权：三个门禁仍分别批准，并各自绑定形成后的真实产物版本。不得预先批准尚未形成的路线或规格；任何模式都不得零确认编码。

## 公开入口边界

Router `grill-harness` 与七个薄入口共享同一状态机。入口完成后只建议下一入口，不得自动串联：`grh-start` 在用户选择路线后停止；`grh-plan` 在最终规格等待批准，批准后生成执行 Frontier 并停止；`grh-run` 不做最终验收；`grh-check` 不实施修复；`grh-recover` 不替用户重新选路；`grh-learn` 不在未批准时写入长期知识；`grh-upstream-check` 始终只读。

## 状态与恢复

合法状态：`pending`、`in_progress`、`needs_user`、`blocked`、`completed`、`stale`、`superseded`、`skipped`、`failed`、`cancelled`。

- `needs_user`：必须等待门禁或产品决策，不能由 Agent 自批。
- `blocked`：外部条件或必需输入缺失；报告解除条件。
- `stale`：上游需求、决策、规格、Git 基线或证据已变化；重新验证受影响链路。
- `superseded`：保留旧产物及替代关系，不删除历史。
- `skipped`：状态值仅为协议兼容保留；当前 v1 的十二个机器阶段全部必需，不得标记为 `skipped`。阶段内部未产生的可选研究/原型产物只记录理由，不改变 phase 状态。

恢复时先 reconcile。出现多工作流、身份冲突、同一产物多个当前版本、Git 基线漂移、手工编辑或证据过期时：保存原文件、不猜权威、不自动修复，列出冲突与候选项，让用户作必要决定。

## 失效传播

需求、约束或决策变化时创建 `CHG-xxx`，将依赖它的路线、研究结论、规格、任务、实施报告、审查和证据标为 `stale`。只重做受影响的最小链路；不得因后续阶段“已经完成”而保留已失效结论。
