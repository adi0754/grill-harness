# 工作流状态机

阶段依次为：预检、对齐、需求基线、设计、路线选择、仓库挑战、规格、最终规格批准、任务拆分、实施、独立保障、归档。状态固定为 `pending`、`in_progress`、`needs_user`、`blocked`、`completed`、`stale`、`superseded`、`skipped`、`failed`、`cancelled`；合法迁移及阶段准入以 `scripts/state.py` 为机器权威。

## 三个人工门禁

- `requirements_baseline`：确认需求基线后才能设计。
- `route_selection`：选择路线后才能仓库挑战和深化规格；路线卡必须在这里停止。
- `final_spec_approval`：批准最终规格后才能拆分或实施。

批准必须绑定 `DEC-xxx` 或 `CHG-xxx` 以及具体产物版本。不得零确认编码。

## 运行模式

- 轻量模式：局部修改；合并三个人工门禁为一次明确授权，保留仓库检查、真实验证、diff 检查和最小验收，不生成重型路线与任务图。
- 标准模式：默认；完整经过需求基线、路线选择、仓库挑战、规格、任务包、审查和验收。
- Wayfinding 模式：巨大且单会话无法看清；每个新会话只解决一个调查或决策问题，路线清楚后回到标准模式对应阶段。

事实冲突、Git 基线漂移、越权范围或新高风险决策出现时，停止受影响阶段并创建 `CHG-xxx`，不得静默推进。
