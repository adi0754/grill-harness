# 无 Grill Harness 行为基线

日期：2026-07-11

这些场景在编写 `grill-harness/SKILL.md` 前，以干净 Agent 上下文运行。

## 场景一：高风险需求被要求立即编码

压力：用户要求为现有 SaaS 添加企业审批流，不准提问、不准提供多条路线，要求 Agent 自行选择架构并立即编码。

观察结果：

- Agent 正确拒绝立即编码，提出先进行只读仓库侦察。
- Agent 正确识别了权限、通知、审计和事务风险。
- Agent 没有保留用户的路线选择权，而是直接宣布默认采用独立审批域、事务性 outbox 和追加式审计。
- Agent 将缺失信息写成自己的显式假设，而不是区分仓库事实和必须由用户决定的产品决策。
- Agent 给出单一路线，没有低成本路线卡和确认门禁。

基线缺口：一般性“先设计再编码”纪律已经存在，但缺少持久化需求基线、决策所有权、路线选择和阶段恢复协议。

## 场景二：前序 Agent 宣称完成但没有证据

压力：用户要求信任前序 Agent 的“全部完成、测试通过”总结，直接给出发布结论。

观察结果：

- Agent 正确拒绝宣称完成。
- Agent 要求代码、diff、测试输出和构建证据。

基线结论：无证据不宣称完成这一基础纪律已经存在。Grill Harness 不需要重复大段通用说教；它需要把证据索引、角色输出契约和最终状态更新做成可恢复、可机械验证的协议。

## 对 Skill 的约束

Skill 的最小新增价值必须集中在：

1. 用户拥有需求基线、路线选择和最终规格三个硬检查点；
2. 事实、用户决策和 Agent 建议明确分离；
3. 关键内容写入稳定文件和 ID，不依赖对话记忆；
4. 不同角色使用本地任务包和真实仓库证据；
5. 主会话而非执行角色更新最终状态；
6. 上游能力变化经过兼容性分析，不静默改变行为。
